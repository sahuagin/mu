## Things noticed but not addressed
- The "is RunMode actually used as a real state machine vs. just a mode flag" question: in this implementation, `Asking` and `Sleeping` exist as variants but are NOT yet driven — the loop's existing ask-shaped state is implicit (`turn_count`, `started_at`, `tool_history`). Phase C (mu-7zn) will need to wire `Sleeping` into a real sub-state of the outer `select!`. Phase B does NOT yet rely on `Asking` being distinct from `Idle`.
- The "goal-status callout" emitted in MaybeFinish may emit `satisfied: false` with `reason: "no goal_status marker; continuing"` on every iteration where the model didn't include a marker. This is verbose for long runs; consider gating the "no marker" callout behind `goal_check_interval` once Phase B lands.
- The continuation `UserMessage` between iterations ("iteration N: continue toward the goal") is a stub-quality prompt — real spec calls for a system message that "pins the goal and the iteration number." Refining this is in scope of mu-3ao but I left it terse; mark as a refinement for next session.
- `extract_goal_status` uses a naive JSON-substring scan: it picks the FIRST `{` and tries shrinking-end substrings to parse as JSON. A more robust parser would handle nested objects correctly. Adequate for FauxProvider tests; revisit if real-model outputs surface ambiguity.
