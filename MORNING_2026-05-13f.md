## Things noticed but not addressed

- **Cross-repo sync (stretch A — note only, no commit in the other repo):** `~/src/personal/mu-aws-sandbox-infra/docs/mu-integration.md` "Proposed shape: typed `Capability` enum" section (lines 18-42) describes an enum-of-variants design that was superseded by the struct-of-axes choice landed here. Recommended update: rewrite that section to reflect `Capability::aws: HashSet<AwsCapability>` and note that `Capability::intersect` is now the broker-pattern primitive. The "Skill activation as the capability handoff point" section (lines 44-58) reads correctly against the new shape.
- **Hash-vs-Eq subtlety in `AwsCapability`:** `Hash` is on name only; `PartialEq` is over both fields. A `HashSet<AwsCapability>` could in principle store two caps with the same name but different policies as distinct elements. The practical invariant "one cap per name" is maintained by `intersect()`'s same-name collapse, but external constructors of `Capability.aws` should treat one-cap-per-name as a discipline. Worth a `debug_assert!` at construction sites once those exist.
- **`session_policy` intersection algorithm:** deferred. When the second AWS variant exists (e.g. `Capability::Fs` per the design sketch in mu-integration.md "Open questions" §5), the policy-intersection algorithm should be designed once and apply uniformly to any axis that carries an inline-policy sub-axis.
- **`Capability::intersect()` is unused at this commit.** The method exists for the broker-pattern but no caller wires it yet. That's intentional (wire surface before behavior, per mu-036 Phase A.2 pattern) — the broker runtime is blocked-by mu-037 (peer.hello + mailbox).
- The `unused import: ToolCall` warning in `crates/mu-core/src/event_log.rs:545` is unrelated to this work; pre-existing and surfacing on every test build.
